Markdown Files

All memory is stored as plain Markdown files on disk. Human-readable, git-trackable, and easy to edit manually. Two locations: data/MEMORY.md (root) and data/memory/*.md (topic-specific).

Source of Truth
🔍

FTS5 Full-Text Search

SQLite FTS5 virtual table indexes all memory chunks. BM25 ranking returns the most relevant results. Zero external dependencies — no embedding APIs, no vector databases.

Search Engine
✂️

Overlapping Chunker

Files are split into ~1,600 character chunks (~400 tokens) with 320 char overlap. Prefers paragraph boundaries to preserve semantic coherence across chunks.

Indexing
👁️

File Watcher

Recursive fs.watch on the data directory with 1.5s debounce. Detects .md file changes and triggers re-indexing. Uses mtime cache to skip unchanged files.

Live Sync
💫

Soul System

Separate from memory — SOUL.md defines personality and behavior. Injected into every system prompt. Live-reloaded on file change. Soul = who I am, Memory = what I know.

Identity
🧠

Reflection & Signals

Operational signals (errors, tool failures, user sentiment, patterns) are captured in SQLite. A reflection daemon runs every 6 hours to analyze and suggest self-improvements.

How Data Moves Through the System

1 ✏️ Write Path — Data Enters Memory

  • Agent writes via write_file tool — creates or updates data/memory/*.md
  • Cron jobs append data (e.g., daily market analysis, product discovery)
  • User requests like "remember this" trigger the agent to persist information
  • Reflection daemon writes self-analysis to reflection_log.md
⬇

2 📑 Indexing Pipeline — Files Become Searchable

  • File watcher detects .md change → 1.5 second debounce
  • Mtime check — skip files that haven't actually changed
  • Chunker splits file into overlapping ~1,600 char segments at paragraph boundaries
  • Old chunks deleted for the changed file (prevents duplicates)
  • New chunks inserted into memory_fts FTS5 table with path + line numbers
  • Deleted files are automatically cleaned from the index
⬇

3 🔍 Read Path — Agent Queries Memory

  • System prompt instructs the agent to search memory proactively
  • memory_search — LLM issues a tool call with a query string
  • Query tokens are sanitized (quoted for FTS5 literal matching)
  • FTS5 MATCH returns results ranked by BM25 score (lower = better)
  • Results include: file path, chunk text, start/end line numbers, relevance score
  • memory_get — reads specific line ranges from disk for full context
⬇

4 🧠 Reflection Loop — Self-Improvement

  • Signals captured during conversations: errors, tool failures, sentiment, unknown requests
  • Every 6 hours the reflection daemon gathers signal summaries
  • Claude analyzes patterns and proposes the single most impactful improvement
  • Good ideas are recorded as evolution suggestions
  • Signals auto-pruned after 7 days to prevent unbounded growth
